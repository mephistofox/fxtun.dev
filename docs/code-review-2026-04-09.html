#01 YooKassa webhook IP-check обходится через X-Forwarded-For
Critical Security
internal/server/payment/yookassa.go:455-461 & internal/server/api/handlers_payment.go:385
r.RemoteAddr перезаписывается RealIP middleware значением из заголовков X-Forwarded-For / X-Real-IP. Проверка IsYooKassaIP(r.RemoteAddr) сверяет уже подменённый IP, а не реальный TCP-адрес отправителя. Атакующий ставит заголовок X-Forwarded-For: 185.71.76.1 (IP из allowlist YooKassa) с любого адреса и полностью обходит проверку.
Последствия: Атакующий отправляет поддельный payment.succeeded webhook с произвольными metadata (user_id, subscription_id) — бесплатная активация подписок без реальной оплаты. Полный обход платёжной верификации.
#02 GetClientIP доверяет X-Forwarded-For/X-Real-IP безусловно
Critical Security
internal/server/auth/middleware.go:244-276
Функция GetClientIP() приоритетно читает заголовки X-Forwarded-For и X-Real-IP без проверки доверенного прокси. Используется для audit logs, сессий, проверки AllowedIPs на API-токенах. Любой клиент может подделать свой IP через HTTP-заголовки.
Последствия: Обход IP-ограничений на API-токенах, подмена IP в аудит-логах (уничтожение forensic-возможностей), потенциальный обход rate-limiting в API-эндпоинтах.
#03 WriteTimeout: 0 отключён на всём HTTP-сервере
Critical Security
internal/server/api/api.go:437
WriteTimeout: 0 отключает таймаут записи для всех подключений, хотя комментарий говорит «disabled for SSE long-lived connections». Это затрагивает каждый эндпоинт: файловые загрузки, вебхуки, обычные API-запросы.
Последствия: Уязвимость к Slowloris-атаке. Клиент удерживает соединение бесконечно, медленно читая ответ — исчерпание горутин и файловых дескрипторов. SSE должен быть на отдельном mux с http.TimeoutHandler.
#04 Статический обменный курс USD/RUB без live-обновления
Critical Bug
internal/server/exchange/exchange.go:1-33
Обменный курс — глобальная переменная var rate float64, устанавливаемая один раз при старте из конфига (exchange_rate: 80.0). Нет механизма получения live-курса от API. Курс USD/RUB колеблется от 60 до 110+.
Последствия: Ежемесячные autopayment через scheduler (ConvertUSDToRUB(plan.Price)) списывают неправильные суммы. При реальном курсе 100 и конфиге 80 — пользователи платят на 20% меньше. И наоборот.
#05 OAuth callback передаёт токены через URL query parameters
Critical Security
internal/client/gui/auth_service.go:470-501
OAuth callback handler читает access_token и refresh_token из r.URL.Query().Get("access_token"). Токены в URL сохраняются в browser history, серверных access logs, утекают через заголовок Referer.
Последствия: Токены доступны в browser history. На shared-машине или при компрометации — полная утечка учётных данных. Нужно использовать POST body или URL fragment (#token=).
#06 Inspector SSE: auth token передаётся в URL query string
Critical Security
internal/client/gui/inspect_service.go:195
Auth-токен передаётся как ?token=%s в URL запроса к SSE-эндпоинту инспектора. Токен оказывается в логах сервера, прокси-логах, истории браузера.
Последствия: Утечка auth-токена через любое логирование URL. Компрометация позволяет полный доступ к данным инспектора и управлению туннелями.
#07 OAuth merge аккаунтов без согласия пользователя
Critical Security
internal/server/api/handlers_oauth.go:183-199, 472-488
При линковке GitHub/Google аккаунта, если OAuth ID принадлежит другому пользователю, вызывается MergeUsers автоматически без подтверждения. Переносятся все API-токены, сессии, домены, подписки, история.
Последствия: Атакующий, контролирующий GitHub/Google аккаунт, ранее привязанный к жертве, запускает auto-merge и получает полный доступ: все токены, сессии, домены, биллинг.
#08 bcrypt молча обрезает пароли длиннее 72 байт
Critical Security
internal/server/api/dto/requests.go:6 & internal/server/auth/password.go:19
DTO разрешает пароли до 128 символов (validate:"max=128"), но bcrypt.GenerateFromPassword хеширует только первые 72 байта. Пользователь думает, что установил длинный пароль, но фактически защищён только первыми 72 байтами.
Последствия: Два разных пароля с одинаковыми первыми 72 байтами дают идентичный хеш — оба допускают вход. Нужно ограничить max=72 или применить pre-hashing через SHA-256.

High

17 issues
#09 Версия клиента сравнивается лексикографически, не semver
High Bug
internal/server/core/server.go:713-728
authMsg.Version < s.cfg.Server.MinVersion использует Go string comparison (лексикографическое). "0.9.0" < "0.10.0" = false, потому что '9' > '1' на позиции 2.
Последствия: Клиент 0.9.0 проходит проверку минимальной версии 0.10.0. Устаревшие клиенты с уязвимостями не блокируются. Нужно golang.org/x/mod/semver.
#10 Session secret не ротируется после успешного JoinSession
High Security
internal/server/core/server.go:817-830
Session secret с TTL 5 минут не инвалидируется после успешного JoinSession. Перехваченный секрет позволяет подключить неограниченное число data-сессий в пределах TTL.
Последствия: Атакующий, перехвативший session secret, добавляет rogue data-сессии к чужому клиенту, потенциально перехватывая или подменяя трафик.
#11 generateID: сниженная энтропия из-за переиспользования байт
High Security
internal/server/core/server.go:1528-1540
9 random bytes генерируют 12 символов через i%len(b): байты 0-2 переиспользуются для позиций 9-11. Символы на позициях 0&9, 1&10, 2&11 всегда идентичны. Плюс bias от %36 на byte range 0-255.
Последствия: Эффективная длина ID = 9 символов вместо 12. Пространство ID значительно меньше ожидаемого, что увеличивает вероятность коллизий и предсказуемости.
#12 IP allowlist обходится через X-Real-IP без trusted proxy
High Security
internal/server/core/middleware_ip.go:53-75
extractClientIP приоритетно читает X-Real-IP перед RemoteAddr. Без trusted proxy атакующий спуфит X-Real-IP: <allowed-ip> и обходит IP-allowlist туннеля.
Последствия: IP-whitelist на туннелях не защищает от атакующих, которые могут подставить нужный заголовок. Контроль доступа по IP фактически не работает.
#13 UDP hash collision: ответы уходят не тому клиенту
High Bug
internal/server/core/udp_manager.go:229-237
hashAddr генерирует 32-bit хеш IP+port. hashToKey — last-write-wins. По birthday paradox: 50% вероятность коллизии при ~65K уникальных адресах. При коллизии ответ уходит не тому UDP-клиенту.
Последствия: В высоконагруженных UDP-туннелях (DNS, игровые серверы) ответы маршрутизируются неправильному клиенту — нарушение корректности и information disclosure.
#14 authLimiters (sync.Map) растёт без ограничений — memory leak
High Bug
internal/server/core/server.go:142, 838-845
Каждый уникальный IP при попытке авторизации создаёт запись SlidingWindow в sync.Map, которая никогда не удаляется. Ботнет с ротацией IP вызовет неограниченный рост памяти.
Последствия: OOM при целенаправленной атаке с миллионов IP. Даже при обычной работе — медленная утечка памяти за недели/месяцы.
#15 Expiration reminder emails отправляются каждый час без дедупликации
High Bug
internal/server/scheduler/scheduler.go:442-475
sendExpirationReminders запускается каждый час (интервал scheduler). Запрашивает подписки, истекающие через 3 и 1 день. Нет дедупликации — каждый час те же пользователи получают те же письма.
Последствия: Пользователь получает ~48 дублей «осталось 3 дня» и ~24 дубля «остался 1 день». Жалобы, блокировка почтового провайдера, репутационный ущерб.
#16 autocert.Cache: неполная реализация, рискует rate-limit Let's Encrypt
High Bug
internal/server/tls/manager.go:280-286
Cache.Get возвращает только cert.CertPEM, игнорируя private key. Put — no-op (return nil). autocert не может кешировать сертификаты и перезапрашивает их при каждом рестарте.
Последствия: Let's Encrypt rate limit: 5 duplicate certificates per domain per week. После нескольких рестартов сервера — невозможность получить новые сертификаты.
#17 Auto-close одного туннеля убивает ВСЕ туннели клиента

Таймер auto-close вызывает c.Close() — закрывает клиент целиком (все туннели, yamux session, reconnect). Должен закрывать только конкретный туннель.
Последствия: Пользователь с несколькими туннелями теряет все активные соединения при истечении auto-close любого одного туннеля.
#18 Inspector: io.ReadAll без лимита на body — OOM
